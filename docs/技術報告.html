架構核心原則：所有功能都以 純 Python + SQLite + numpy 實作，零外部 AI 模型依賴。向量搜尋用自製 TF-IDF 而非 sentence-transformers，確保離線可用、體積小、啟動快。
⚙

技術棧 & 整體架構

🗃️
持久層 — SQLite WAL Mode
所有資料儲存在 ~/.get-to-it.db，使用 WAL（Write-Ahead Logging）模式，支援並發讀取不阻塞寫入。

Migration 使用冪等模式：所有新欄位以 try/except + ALTER TABLE 實作，`init` 可反覆執行不損壞現有資料。
🐍
運行時 — Python 3 單檔架構
整個系統是一個 2,429 行的單一 gti.py 腳本，無框架依賴。所有命令透過 argv 解析，輸出統一為 JSON（除 telegram 格式外）。

可選依賴（啟動時 try/import，缺少時優雅降級）：numpy、icalendar、urllib
📦
Cowork Skill 封裝
SKILL.md（AI 行為規則）+ scripts/gti.py（CLI 工具）+ references/persona.md 打包成 .skill zip。

Claude 讀取 SKILL.md 後決定何時呼叫 gti.py、如何解讀 JSON 輸出、以什麼語氣回應使用者。
🔢
向量層 — 純 numpy TF-IDF
不使用任何預訓練模型。自製字元 bigram/trigram + 詞彙 token TF-IDF，用 numpy 計算 cosine similarity。

向量以 JSON 存入 SQLite long_term_memory 表。個人規模（<1000 筆）的線性掃描速度足夠快，且天然支援中文（字元 n-gram 不需斷詞）。
為什麼不用 sentence-transformers？ 初始嘗試時因磁碟空間不足（CUDA 套件約 350MB）失敗。改用純 numpy 方案後，在個人使用規模下效果同樣良好，且啟動時間從 3-5 秒降至毫秒級。

V2 P1

Priority 1 — 時間追蹤 & 行事曆整合

讓系統知道「這件事你通常花多久」以及「今天行程表有什麼」

⏱️
任務計時追蹤
start TASK_ID 記錄 started_at，complete 時自動計算 actual_minutes（或 --minutes N 手動覆蓋）。

time_source 欄位記錄來源：auto（系統計時）或 manual（使用者手填），用於校準信度權重。
📐
EWMA 時間預測校準
每次完成任務後計算 ratio = actual / estimated，以 EWMA（α=0.3） 更新 time_models 表的 rolling_ratio。

smart-estimate CATEGORY MINUTES 根據歷史比值自動校準估時（例如：你在 coding 類通常低估 1.4 倍，系統幫你放大）。
📅
iCal 行事曆整合
connect-calendar NAME URL --type ical_url|ical_file 儲存訂閱連結到 calendar_sources 表。

支援 Google Calendar、Apple Calendar 的 .ics 格式。Morning brief 自動呼叫 sync-calendar 撈取當天事件，解析 VEVENT 並處理 timezone/all-day event 差異。
🕐
空閒時間計算
free-time 讀取今日 calendar_events，從預設工作時段（09:00-18:00）扣除行程，計算剩餘可用小時。

Morning brief 若行事曆已連結，自動使用計算出的 free_time 作為 available_hours，取代手動輸入。
新增 CLI 指令：time-stats、smart-estimate、connect-calendar、sync-calendar、list-calendars、disconnect-calendar、free-time

新增資料表

資料表用途關鍵欄位
time_modelsEWMA 校準模型（per category）rolling_ratio, sample_count, last_updated
calendar_sources已連結行事曆清單name, source_path, source_type, enabled
calendar_events已同步的行事曆事件summary, dtstart, dtend, event_date, source_id
daily_logs每日工時記錄date, available_hours, tasks_completed
技術細節：ical timezone 處理 — ical 事件的 dtstart 可能帶有 tzinfo（datetime）或為純日期（date）。程式碼特別處理：datetime 物件先 stripping tzinfo 轉成 naive datetime，date 物件（全天事件）轉為當天 00:00-23:59 區間，避免比較時的 TypeError。
V2 P2

Priority 2 — 向量長期記憶 & Agent 失敗處理

讓系統永久記住重要的個人資訊，並自動偵測卡住的 Agent 任務

🧬
自製 TF-IDF 向量引擎
_tokenize(text)：字元 bigram + trigram + 空白分詞，支援中文無需斷詞器。
_build_vocab_and_vectors(texts)：建立 IDF 字典（log(N/df+1)）和 TF 矩陣。
_text_to_vector(text, vocab_idx, idf)：將查詢字串投影到相同空間。

Cosine similarity 搜尋，個人規模（<1000 筆）線性掃描，毫秒內完成。
零模型下載中文友善numpy only
💾
長期記憶 CRUD
store-ltm：儲存重要陳述，計算並存入 vector（JSON float array）。
recall-ltm：語意搜尋（cosine similarity top-N），去掉向量欄位再回傳。
list-ltm：列出所有記憶，可按 type 過濾。
clear-ltm：刪除單筆。

Morning brief 用今天任務標題查詢 recall_relevant，結果放入 long_term_context 欄位。
語意搜尋永久儲存
🤖
Agent 超時偵測
agent-status 掃描所有 agent_processing 任務，計算距 updated_at 的分鐘數。超過 30 分鐘自動標記為 agent_failed。

回傳 needs_attention[] 清單，morning brief 在 Top 3 之前先呈現需要注意的 Agent 任務。
30 分鐘門檻自動偵測
🔁
Agent 任務恢復
agent-retry：將 agent_failed 任務重設為 agent_processing，更新 updated_at，供 Claude 重新派發。

agent-handoff：將 Agent 任務改回 assignee=human，移出 AI 執行佇列，交由使用者自行處理。

兩個動作都記錄到 action_log 留存稽核軌跡。
retryhandoff

V2 P3

Priority 3 — ML 權重自動調整 & Telegram 整合

讓系統從你的行為中學習，並支援透過 Telegram 傳送晨間簡報

📊
推薦紀錄 Logging
每次 morning brief 將 Top 3 任務的詳細因素貢獻記錄到 recommendation_log：

· task_id、brief_date、position（1-3）
· priority_score（總分）
· factors（JSON，每個因素的分數）
· was_completed（0/1）、completed_at

當使用者 complete 任務時，自動 UPDATE was_completed=1。
⚖️
動態評分權重 DB
scoring_weights 表儲存六個因素的當前權重，含 sample_count 和 last_trained。

預設種子值：deadline 30.0、goal_priority 20.0、momentum 15.0、time_fit 15.0、skip_penalty 10.0、proximity 10.0。

load_scoring_weights() 從 DB 讀取，提供給 morning_brief 評分使用。_source 欄位標示 "default" 或 "learned"。
🎓
Logistic Regression 權重學習
learn_weights()：從最近 90 天 recommendation_log 取出資料，建立 feature matrix X（各因素貢獻正規化後的值）和 label vector y（was_completed）。

純 numpy gradient descent（500 次迭代，lr=0.1），收斂後將 |θ| 正規化到總預算 100 分，寫回 scoring_weights 表。需要 ≥10 筆樣本。
純 numpy無 sklearn≥10 樣本
📱
Telegram MarkdownV2 格式
morning --format telegram 呼叫 _format_morning_telegram()，輸出 Telegram MarkdownV2 格式的文字（非 JSON）。

包含：日期 header、可用時間、行事曆事件數、Agent 注意清單、Top 3 + 圖示 + 時間估計、長期記憶摘要、總估時。

括弧 () 等特殊字元以 \(...\) 跳脫，符合 Telegram 規格。

關鍵除錯記錄

開發過程中遇到的所有非預期問題及修復方式

BUG #1
sqlite3.Row 不支援 .get()
complete_task() 中用 task.get("category") 觸發 AttributeError。
修復：改用 try/except 包裝 task["category"] 直接存取。
BUG #2
向量帶入 recall 回傳結果
long_term_memory 的 vector 欄位（JSON 數百 float）出現在搜尋結果中。
修復：recall_relevant() 回傳前加入 r.pop("vector", None)。
BUG #3
孤立的 db.execute 殘留
TF-IDF fallback 路徑中有一行裸露的 db.execute 呼叫（沒有接收回傳值）。
修復：移除該行。
BUG #4
needs_attention 為空（Agent 超時偵測）
get_agent_status() 在 DB 更新後，in-memory 的 results 清單狀態仍為 agent_processing，導致 needs_attention 過濾不到任何項目。
修復：在更新 DB 的同時也更新 in-memory dict：t["status"] = "agent_failed"。
BUG #5
Agent 超時測試的 UTC/Local 時區不一致
SQLite datetime('now', '-35 minutes') 以 UTC 計算，Python datetime.now() 以本地時間計算，造成時差導致測試失敗。
修復：測試改用 (datetime.now() - timedelta(minutes=35)).strftime(...)，確保時區一致。
BUG #6
sentence-transformers 安裝失敗（改用純 numpy）
磁碟空間不足，CUDA 相關套件約 350MB 無法下載。
修復：放棄 sentence-transformers 和 chromadb，改用自製 TF-IDF + numpy cosine similarity，個人規模下效果相當，零額外依賴。
BUG #7
ical proxy 403 錯誤
嘗試在沙盒環境中連接外部 .ics URL 時，遭 proxy 封鎖（HTTP 403）。
修復：改以 --type ical_file 搭配本地 .ics 檔案路徑做測試驗證；正式環境由使用者在本機執行不受限。
BUG #8
morning_brief 中的死代碼（weights_trained）
曾有一段 if False else False 的死代碼嘗試查詢 scoring_weights（但 db 已關閉）。
修復：整段移除，weights_source 改從 load_scoring_weights() 返回值取得。

已知限制 & 未來可能方向

⚠️
TF-IDF 在大量語料下退化
當 long_term_memory 超過 ~2000 筆時，全量線性掃描開始變慢，詞彙矩陣也可能過大。

可能升級路徑：改用 FAISS 或 sqlite-vss 向量索引，或在累積到一定量時才啟用近似搜尋。
⚠️
ML 訓練需時日積累
learn-weights 需要 ≥10 筆，但訓練效果要到 ~30 筆後才有意義，~100 筆後才穩定。

可能改善：初期用 Bayesian prior（預設權重作先驗），避免樣本少時過擬合。
⚠️
單檔架構可維護性
2,429 行單一腳本在此規模仍可維護，但繼續增長後會變難。

可能重構：拆分 db.py、scoring.py、calendar.py、memory.py 模組，保持 gti.py 作為 CLI 入口。
